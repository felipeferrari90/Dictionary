REST é um padrão de API (tem o padrão SOAP), é um protocolo que age no HTTP digamos assim é o nivel de maturidade em que sua API se encontra, REST pode se dizer que é um medidor funcional da sua API e REST são niveis desse medidor -niveis de maturidade do richardson -> LEVEL 0: (the swamp of fox) -é o nivel mais comum de uma API no mercado onde vc usa 1 ENDPOINT ou poucos para varias features, EX: vc constroi uma api pra uma loja, e entao vc usa o endpoint dela para cadastar vendas. cadastrar distribuidor, cadastrar categorias e etc. -> LEVEL 1: (resources) -aqui vc ja tem um endpoint para cada feature EX: sua mesma api do mercado agora tem um endpoint para cliente, um end point para produto, um para fornecedor, vendas etc, é um endpoint para cada recurso especifico. -> LEVEL 2: (verbs http) -alem de um endpoint para cada recurso se usa os verbos http neles(GET,POST,PUT,DELETE etc) -um mesmo endpoint com métodos diferentes -EX: existiria endpoints com methods POST para criar mercadoria um GET para listar mercadoria ai existiriam endpoints de criacao e exclusao de clientes, fornecedores e etc. tudo gira em torno de endpoints para CRUD REST FUL -> LEVEL 3: (HATEOAS - hypermedias controls) - é quando URLS são retornadas como link atraves do JSON, nesse nivel podemos dizer que sua api é REST FULL EX: imagine a api da sua loja devolvendo o JSON em produtos,(sem HATEOAS) { "nome": "Caderno", "descricao": "Caderno Espiral 100 folhas", "preco": 5.45 } agora imagine ela devolvendo o mesmo produto porem ele contendo um link onde você pode acessa-lo, esse link é uma hypermedia e nesse caso estamos trabalhando com o modelo HATEOAS { "nome": "Caderno", "descricao": "Caderno Espiral 100 folhas", "preco": 5.45, "links" : [{ "rel" : "self" "href" : "http://localhost:8080/produto/1" "method":"GET" }, "links" : [{ "rel" : "update_produto" "href" : "http://localhost:8080/produto/1" "method":"PUT" }, "links" : [{ "rel" : "delete_produto" "href" : "http://localhost:8080/produto/1" "method":"DELETE" }] } rel - Significa relacionamento. No nosso caso o link autoreferencia o produto. Sistemas mais complexos incluem outros tipos de relacionamentos; href - caminho que leva ao produto -> LEVEL 3+ : (a glory of rest) - após isso vc pode adicionar mais coisas, paginacao, otimizacao, mas ela ja é REST de qualquer jeito. O objetivo da URL é associar um endereço remoto com um nome de recurso na Internet. URN – Uniform Resource Name - Nome de Recursos Universal é o nome do recurso que será acessado e também fará parte da URI. home.html contato.php servicos.html É comum associarmos URN a página que estamos acessando. ex.: /api/v01/exemplo/usuarios numeros ISBN são os melhores exemplos de URN url para lei seca no brasil ex.: urn:lex:br:federal:lei:2008-06-19;11705 resolver - os resolvers de URN é responsavel por interpretar um dado URN traduzindo para URL Resolvedores locais, centrais e federados: a resolução de URNs é idealmente realizada por um PURLs (Persistent Uniform Resource Locators)[3], mas pode ser também uma ferramenta para portais locais. Uma forma de garantir a consistência e operacionalidade entre resolvedors locais e centrais é a federação: o URN resolver local quando não sabe resolver, repassa o pedido para o central. Pode ser uma imagem, uma página, etc, pois tudo o que está disponível na internet precisa de um identificador único para que não seja confundido.
